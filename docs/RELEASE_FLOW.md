# 릴리즈/운영 반영 절차 (Smartlog AI)

## 브랜치 규칙(권장)
- `main`: **운영 배포용** (항상 배포 가능 상태 유지)
- `dev`: **개발 통합** (기능 개발이 합쳐지는 곳)
- `feature/*`: 기능/버그 단위 작업 브랜치

## Netlify 연결(권장)
- 운영(Netlify Production): `main`만 배포
- 스테이징(Netlify Staging 또는 Deploy Preview): `dev` 또는 PR Preview 배포

## 릴리즈 단위(운영 반영)
1) `feature/*` → `dev`로 머지\n
2) 스테이징에서 기능 검증\n
3) 운영 반영 시점에 `dev` → `main`으로 릴리즈 머지\n
4) 운영 URL에서 핵심 플로우 점검(로그인/업무기록/승인/자료실)\n

## 운영 안정성 수칙(중요)
- 운영 배포는 “자주”가 아니라 “안전한 단위”로 묶어서 진행\n
- DB 스키마 변경은 `docs/sql/*.sql`로 관리하고, 적용 이력(날짜/담당/적용 대상)을 남김\n
- 운영/스테이징 Supabase는 반드시 분리\n

## 버전 표기(권장)
- `main.html`의 `?v=YYYYMMDDx`를 릴리즈 시점에 갱신\n
- 장애/피드백 시 “어느 버전에서”인지 빠르게 식별 가능\n
